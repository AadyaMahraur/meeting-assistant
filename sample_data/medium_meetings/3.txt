INCIDENT COMMAND WAR ROOM: SEV-1 Outage "The Double Dip"
Date: Friday, November 13th (Of course it is)
Time: 08:00 AM PST (Incident started at 02:14 AM PST)
Present: Sarah (VP Eng), Ben (Lead Backend), Raj (DevOps/SRE), Chloe (Head of Support)
Status: DEFCON 1. Payments system double-charged ~5,200 customers. Zendesk is literally crashing.

[08:00] THE "WHAT THE HELL JUST HAPPENED" TIMELINE
Sarah: Nobody is sleeping until this is fixed and every cent is refunded. Raj, give me the timeline. No sugar-coating.

Raj: * 02:14 AM: AWS us-east-1 had a network blip. Our primary Aurora Postgres instance initiated an automated failover to the read-replica.

02:14 - 02:15 AM: Database was locked/unreachable for exactly 48 seconds during the DNS propagation.

Ben: During that 48 seconds, the overnight ACH batch processing job kicked off.

Raj: Right. The Celery workers picked up the batch file (about 12,000 pending subscriptions). They tried to write the "processing" state to the DB. DB timed out.

Ben: And this is where it gets bad. The workers caught the OperationalError, and triggered the built-in retry mechanism. Exponential backoff, max retries = 3.

Sarah: But if they retried, why did they double-charge? Doesn't Stripe have idempotency keys to prevent exactly this?

Ben: [Long pause] Yes. Stripe does. But... our idempotency key generation logic was flawed.

The Root Cause (Whiteboard diagram transcribed):

How it should work: Idempotency_Key = Hash(User_ID + Invoice_ID). If you send it twice, Stripe ignores the second one.

How we built it: Idempotency_Key = Hash(User_ID + Timestamp_of_Request).

The Result: When the Celery worker retried 30 seconds later, the timestamp changed. Stripe saw it as a completely new, valid charge.

Conclusion: We successfully charged 5,200 people. Then the DB came back online, the retry succeeded, and we charged them again.

Ben is writing a Python script to pull all duplicate charge_ids from the Stripe API created between 02:14 and 02:20 AM.

Rule: We DO NOT query our own database to find the duplicates, because our database state is currently untrustworthy due to the failover timeout. Stripe is the source of truth right now.

Rate Limiting Strategy: Batching requests. 50 refunds per second. time.sleep(1) between batches.

WARNING from Raj: "Ben, do a dry-run first. Print the Charge IDs to a text file. If you accidentally refund the original charges instead of the duplicates, we are going to lose $750,000 in revenue."

Priority 3: The Code Patch (Fixing the Idempotency)

Ticket created: ENG-992: Refactor Stripe Idempotency Key Generation

New Logic: Key = UUID(Invoice_ID). Full stop. No timestamps. It must be deterministic.
