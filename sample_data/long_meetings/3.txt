[09:15] DATABASE SCHEMA NIGHTMARE (Taylor & Sam yelling at each other)
•	Topic: Multi-tenancy approach.
•	Options on the board:
1.	Silo: Separate DB instance per client. (Sam: "Are you insane? Managing 50 RDS instances? The Terraform will be 10,000 lines long.")
2.	Bridge: One DB, separate schemas per client. (Taylor: "Postgres schemas are fine but migrations are a nightmare when we have 100+ clients. client_a.users, client_b.users... gross.")
3.	Pool: One DB, one schema, tenant_id on EVERY SINGLE TABLE.
•	DECISION: POOL MODEL.
o	Rules: Row Level Security (RLS) in Postgres is MANDATORY. If someone forgets a WHERE tenant_id = X clause in the ORM, we breach data privacy and get sued into oblivion.
o	Sam: "We need a custom linter rule that blocks any PR that doesn't include tenant context in the DB query."
o	Taylor: "We are ditching Sequelize. It's too bloated. Moving to Prisma or Drizzle. Drizzle is closer to SQL, better for strict RLS policies." -> Action: Taylor to spike Drizzle ORM + Postgres RLS this weekend.
•	Core Tables Draft (Very rough):
o	tenants: id (UUID), name, slug, tier (enum: basic, pro, enterprise), settings (JSONB - keep this lightweight, no deep nesting!)
o	users: id, tenant_id (FK), email, password_hash, status.
o	roles: id, tenant_id, name (e.g., "Warehouse Manager", "Admin").
o	permissions: id, resource, action (e.g., inventory:write, reports:read).
o	role_permissions: linking table.
o	inventory_items: id, tenant_id, sku (MUST BE UNIQUE PER TENANT, NOT GLOBALLY), name, qty_on_hand, location_id.
[10:00] AUTHENTICATION & AUTHORIZATION (The "Don't Roll Your Own Crypto" talk)
•	Currently using some janky custom JWT implementation. We need Enterprise SSO (SAML, OIDC) or we can't sell to anyone bigger than a mom-and-pop shop.
•	Jess: "If we use Auth0, we pay out the nose per MAU."
•	Taylor: "B2B has low MAU but high value. Auth0 pricing is fine for B2B. BUT what about WorkOS? WorkOS is literally built for adding Enterprise SSO."
•	RABBIT HOLE: RBAC vs ABAC.
o	Role-Based Access Control is easy. (Admin can do X, User can do Y).
o	Attribute-Based Access Control is hard. (User can edit Inventory Item ONLY IF Location = "New York" AND Item_Type = "Hazardous").
o	Mark: "Do we need ABAC for MVP?"
o	Silence in the room.
o	Decision: Stick to RBAC for V1. Hardcode permissions.
•	Whiteboard diagram for Auth Flow:
1.	User hits /login -> redirects to Identity Provider (Okta/Azure AD via WorkOS).
2.	Callback hits our backend -> We verify token, find/create User, map SSO groups to our local roles.
3.	Generate local session.
4.	WAIT: Are we doing JWTs in local storage or HttpOnly cookies?
5.	Jess: "HttpOnly cookies. I am not dealing with XSS token theft in an enterprise app. We use a reverse proxy to handle the cross-domain cookie issues if UI and API are on different subdomains."
6.	Taylor: "Fine. API is api.chimera.app, UI is app.chimera.app. We set cookie domain to .chimera.app."

•	"The current React SPA is a 14MB bundle of garbage. I'm throwing it out."

o	Remix: Better data loading, relies on web standards, great for form-heavy B2B dashboards.
o	Next.js (App Router): Server components are cool but the caching model is confusing as hell and might break our real-time inventory updates.

•	State Management:
o	Zustand for global UI state (sidebar open/close, dark mode).
o	TanStack Query (React Query) for ALL server state.
o	Rule: NEVER put API responses in Zustand. Let React Query handle caching, invalidation, and background refetching.
•	The "Massive Table" Problem:
o	Inventory view might have 50,000 rows.
o	Pagination vs Infinite Scroll?
o	B2B users hate infinite scroll. They want page numbers and deep linking. (e.g., "Go to page 4 and look at row 12").
o	URL-driven state is mandatory. /inventory?page=4&sort=sku_asc&filter_loc=NY.
o	If you refresh the page, it MUST stay exactly where you were.
o	Tech required: @tanstack/react-table for headless UI, virtualization (react-virtual) if we decide to do long lists later, but start with standard cursor-based pagination.

[13:00] THE "OH CRAP" MOMENT - AUDIT LOGS
•	Sam: "I just looked at the design partner requirements. AlphaCorp requires an immutable audit log for every single inventory mutation."
•	Dead silence.
•	Taylor: "Every. Single. Mutation?"
•	Sam: "If a warehouse guy changes stock from 10 to 9, we need to know who, when, IP address, and previous state."
•	How to implement?
1.	Application level: Write to an audit_logs table manually on every API route. (Error prone, someone will forget).
2.	Database level: Postgres Triggers. (Hard to maintain, keeping track of the "acting user" is tricky inside a DB trigger).
3.	Event Sourcing. (Taylor: "Absolutely not. We are not building an event sourcing system in 45 days. I will quit.")
•	Compromise: We use Drizzle ORM middlewares/extensions. Intercept all update and delete queries, extract the diff, and push a payload to the BullMQ queue to asynchronously write to an audit_logs table.
•	Schema for audit_logs: id, tenant_id, user_id, table_name, record_id, action (INSERT/UPDATE/DELETE), old_data (JSONB), new_data (JSONB), timestamp.
•	Warning: This table will grow to billions of rows. We must partition it by month immediately. Postgres native partitioning on created_at.
